# AI Communication Studio Workflow

## Рабочий процесс

1. Прочитать корневой `AGENTS.md` и выполнить его требования.
2. Проверить `git status` и не затрагивать несвязанные изменения.
3. Зафиксировать названные пользователем раздел, продукт, экран, блок и сценарий как закрытую область задачи; определить минимальный список файлов для изменения.
4. Найти экран через `src/features/mvp/mvp-shell.tsx`, затем прочитать связанный screen, store, repository, schema и тесты.
5. Сформулировать проверяемые критерии готовности из запроса пользователя.
6. Внести минимальные изменения только внутри зафиксированной области.
7. Добавить или обновить тесты пропорционально риску.
8. Выполнить проверки из раздела «Контроль качества».
9. Деплоить только по отдельной команде пользователя и после деплоя проверить production URL.

## Правила проекта

- Строго выполнять только поставленную задачу. Не расширять её область самостоятельно.
- Считать изменение любых неназванных разделов, продуктов, экранов, блоков и работающих сценариев запрещённым, включая рефакторинг, форматирование и попутные исправления.
- Если необходима правка общего контракта или файла за пределами области задачи, остановиться до изменения, описать зависимость и запросить отдельное явное разрешение пользователя.
- Открывать экраны главного приложения через `/?view=<view>`.
- Использовать компоненты из `src/shared/ui` и текущие UX-паттерны.
- Хранить данные MVP через repository и Zustand stores; не читать Local Storage напрямую из экранов, если уже существует store/repository.
- При изменении Zod-схем сохранять совместимость с legacy snapshot. Сначала нормализовать старые данные, затем валидировать; некорректные необязательные элементы безопасно отбрасывать, а критические данные сохранять в backup.
- Согласовывать историю запусков между Песочницей, Дашбордом и Оценкой Summary.
- Не менять соседние разделы для косметического улучшения и не выполнять несвязанный рефакторинг.
- Не хранить API-ключи, `.env` и пользовательские данные в Git.

## Контроль качества

Выполнять проверки в таком порядке:

1. Запустить узкие тесты изменённого модуля.
2. Запустить `npm test`, если затронуты repository, store, schema или общий контракт.
3. Запустить `npm run lint`.
4. Запустить `npm run build`.
5. Для UI запустить `npm run test:smoke`.
6. Визуально проверить изменённый сценарий в браузере на desktop и mobile viewport.
7. Проверить отсутствие runtime overlay, `pageerror`, ошибок console и перекрытий интерфейса.
8. Просмотреть `git diff --check` и итоговый diff; подтвердить отсутствие изменений в других разделах, продуктах, экранах и блоках.

Не сообщать «готово», пока обязательная проверка не прошла. Если проверка недоступна или падает по внешней причине, указать точный пробел и его влияние.

## Проверка ключевых экранов

Использовать `npm run test:smoke` для основных разделов:

- `product`
- `playground`
- `summary-review`
- `summary-report`
- `dashboard`
- `settings`

Для изменённого пользовательского потока дополнительно выполнить действия вручную: ввод, запуск, переход, сохранение, возврат и повторное открытие данных. Не считать одну загрузку страницы достаточной проверкой интерактивного сценария.

## Деплой

Перед деплоем повторить unit, lint, build и smoke-проверки. После успешного production deployment открыть публичный URL, проверить изменённый сценарий и ошибки браузера. В финальном сообщении указать production URL и фактически выполненные проверки.
