# AI Communication Studio

## Общение и границы задачи

- Всегда отвечай пользователю на русском языке.
- Перед изменениями прочитай связанные компоненты, хранилища, схемы и тесты.
- Работай строго только в рамках задачи, которую поставил пользователь. Считать названный раздел, продукт, экран, блок и пользовательский сценарий закрытым списком разрешённых изменений.
- Жёстко запрещено изменять, редактировать, рефакторить, форматировать или «улучшать заодно» любые другие разделы, продукты, экраны, блоки и работающую функциональность, даже если в них обнаружены ошибки или возможные улучшения.
- Не выполнять несвязанные обновления зависимостей, переименования, переносы файлов, массовое форматирование и очистку кода.
- До начала правок определить минимальный список файлов, необходимых для задачи. Перед завершением сверить итоговый diff с этим списком и удалить только свои случайные изменения вне области задачи, не затрагивая пользовательские изменения.
- Если задачу технически невозможно выполнить без изменения общей части или файла за пределами указанной области, остановиться до такой правки, объяснить пользователю точную зависимость и получить отдельное явное разрешение. Самостоятельно расширять область задачи запрещено.
- Сохраняй пользовательские и незавершённые изменения в рабочем дереве. Не откатывай файлы, которые не относятся к задаче.
- Не выполняй деплой без отдельной команды пользователя.

## Структура приложения

- Основной интерфейс находится в `src/features/mvp` и открывает экраны через `/?view=<view>`.
- Данные MVP хранятся в Local Storage через `src/shared/repositories/local-storage-repository.ts` и Zustand stores в `src/shared/stores`.
- Изменения схем храним обратно совместимыми. При ошибках Zod сначала проверяй миграцию legacy-данных и содержимое Local Storage.
- История запусков Песочницы хранится в playground test run store и должна согласованно отображаться в Песочнице, Дашборде и Оценке Summary.
- Общие UI-компоненты переиспользуй из `src/shared/ui`; новые визуальные паттерны добавляй только при отсутствии подходящего существующего компонента.

## Обязательная проверка

Перед сообщением «готово»:

1. Сверь результат с каждым требованием пользователя и подтверди, что итоговый diff не содержит изменений других разделов, продуктов, экранов или блоков.
2. Запусти релевантные unit-тесты, затем `npm test` при изменении общих контрактов или хранилищ.
3. Запусти `npm run lint` и `npm run build`.
4. Для UI-изменений запусти `npm run test:smoke` и визуально проверь изменённый сценарий в браузере на desktop и mobile viewport.
5. Убедись, что нет runtime overlay, ошибок страницы и ошибок console, а текст и элементы не перекрываются.
6. После деплоя отдельно проверь production URL тем же пользовательским сценарием.

Если какую-либо проверку выполнить нельзя, не называй работу готовой: явно укажи, что именно не проверено и почему.

## Команды

- Локальная разработка: `npm run dev`
- Unit-тесты: `npm test`
- Линт: `npm run lint`
- Production-сборка: `npm run build`
- Smoke-тесты ключевых экранов: `npm run test:smoke`
- Установка браузера Playwright после первой установки зависимостей: `npx playwright install chromium`
